4. Automated vs manual

Automated (stream / scheduler)

  • Gate: ~10 new responses (stream)
  • May cache-hit if nothing new to process
  • Report regen only if 25+ new since last report
  • Incremental sample (~50 new for ABSA)

Manual (Generate button)

  • Gate: 10+ new since last run, max 3/day
  • Always re-narrates and regens report
  • Full bootstrap sample (highest quality)
  • force_regenerate = true

9. Crystal chat vs insight pipeline

Pipeline WRITES

  • Stream / scheduler / manual Generate
  • Creates insights, snapshots, checkpoints

Crystal READS

  • get_metric_history
  • get_insights_list
  • get_topic_details
  • get_checkpoint_history

Crystal streaming = how answers appear in the UI (SSE).
Response stream = how new survey responses trigger insight generation (Redis).
These are different things. Crystal does not write checkpoints.

10. Data safety

Safe by design

  • insights: supersede → one active truth
  • responses: never mutated by pipeline
  • Anchor run prevents manual narrative drift

Noisy (not corrupt)

  • Checkpoints append every run
  • delta_from_prior often null
  • Cache-hit runs still add snapshots

Gaps vs design docs

  • compute_delta not wired in pipeline
  • 200-response checkpoint gate unused
  • schedule trigger may fail checkpoint DB constraint

11. Operator cheat sheet

WHEN WILL INSIGHTS AUTO-UPDATE?
  • 10 new responses (prod) → stream fires pipeline
  • Milestones 10 / 40 / 70 / 100 → extra progressive run
  • Scheduler → backup if no run in last hour

WHEN WILL NARRATIVE REPORT REGEN? (automated only)
  • Only when 25+ new responses since last report.* insight

WHEN CAN ANALYST FORCE REFRESH?
  • Generate button: 10+ new since last run, max 3 per day
  • Always full re-narration + report

WHERE TO LOOK FOR TRENDS
  • Charts / Crystal → survey_metric_snapshots
  • NOT primarily checkpoint blobs (today)

ANOMALIES TODAY
  • Volume spike: 3-day avg > 2x the 30-day average
  • NPS drop → configure alert rules (if enabled)

Part B — Engineering deep dive

13. Critical model: two parallel histories

Engineers must separate live insight state from checkpoint archive. They are written in the same node_publish but consumed differently on the next run.

StoreWritten whenRead by next pipeline run?Read by Crystal?
insights (active rows) Every publish (supersede + insert) Yes — via anchor run_id Yes — get_insights_list
survey_metric_snapshots Every publish Yes — get_prior_metric_snapshots(limit=5) in node_ingest Yes — get_metric_history
survey_insight_checkpoints + blob Every publish (if INSERT succeeds) No — not loaded in insights.py Yes — get_checkpoint_history (metadata only today)
agent_runs audit fields Every publish Yes — anchor selection SQL Indirect
Implication: Creating a checkpoint does not directly change how the next insight pipeline narrates. The next run uses insights rows from the anchor run and survey_metric_snapshots — not checkpoint JSON blobs. Checkpoints are an append-only audit trail + Crystal/history API surface.
What checkpoints indirectly affect: Crystal answers when the model calls get_checkpoint_history; future work if compute_delta() is wired into node_publish and delta_from_prior is populated.

15. How checkpoints are created (exact code path)

Runs inside node_publish after insights are committed. No gating on response count, cache hit, or trigger type.

Algorithm

# node_publish — Phase 2 (after insight transaction committed)

1. checkpoint_number = MAX(checkpoint_number) + 1
   FROM survey_insight_checkpoints
   WHERE survey_id = ? AND org_id = ?

2. report_blob = {
     schema_version: 1,
     survey_id, org_id, checkpoint_number,
     response_count: len(state.responses),      # loaded sample size, not necessarily all DB rows
     nps, csat, ces: from state.metrics,
     insights: [{id, headline, layer}] × 50,    # truncated summary
     topics:   [{name, volume}] × 20,
     metrics:  full metrics dict,
     delta:    null,                             # compute_delta() NOT called
     generated_at: UTC ISO timestamp
   }

3. checkpoint_id = f"ckpt-{run_id}"
4. blob_ref = await write_checkpoint_blob(report_blob, org_id, survey_id, checkpoint_id)
   # dev:  /tmp/checkpoints/{org_id}/{survey_id}/ckpt-{run_id}.json
   # prod: OCI object key checkpoints/{org_id}/{survey_id}/ckpt-{run_id}.json

5. INSERT INTO survey_insight_checkpoints (
     survey_id, org_id, checkpoint_number, trigger,
     response_count_at_checkpoint,
     nps_at_checkpoint, csat_at_checkpoint, ces_at_checkpoint,
     report_url,    -- stores blob_ref (path or OCI key)
     schema_version
   )

crystalos/graphs/insights.py ~L3554–3605 · crystalos/lib/checkpoint_store.py

DB row vs blob

survey_insight_checkpoints columnSourceUsed on next run?
checkpoint_numbermonotonic per survey+orgNo
triggerstate.trigger (stream/schedule/manual)No
nps_at_checkpoint etc.metrics at publish timeNo (snapshots table used instead)
report_urlblob refOnly via REST/Crystal tools
delta_from_priornever populated in pipelineNo
topic_fingerprintnever populated in pipelineNo

Failure modes

  • Exception → logged as checkpoint_write_failed; insights remain published (Phase 2 is non-transactional with Phase 1).
  • trigger='schedule' may violate DB CHECK constraint (responses|days|manual|stream) — row insert fails silently in try/except.
  • CHECKPOINT_FULL_RESPONSE_THRESHOLD=200 in constants is not consulted before write.

17. What feeds the NEXT run (anchor model — not checkpoints)

Incremental narration uses prior_insights loaded in node_narrate from the anchor run in agent_runs + insights.

Anchor run selection SQL

SELECT id FROM agent_runs
 WHERE survey_id = ? AND org_id = ? AND status = 'completed'
   AND run_type = 'insight_generation' AND id != current_run_id
   AND (
         trigger_type IN ('schedule', 'stream')
      OR new_response_count > 0
   )
 ORDER BY completed_at DESC
 LIMIT 1

Then load all insights where insights.run_id = anchor_run_id (not superseded filter — historical run rows).

Prior insight selection

PRIOR_INSIGHT_MIN_TRUST = 65
PRIOR_INSIGHT_MAX_COUNT = 8
PRIOR_INSIGHT_LAYERS    = ("prescriptive", "diagnostic")

preferred = anchor insights matching layer + trust >= 65, up to 8
→ injected into specialist_overlay as "ESTABLISHED FINDINGS" block
→ passed to narrate_topic_insight, prescriptive, trend experts, etc.

18. How new insights are calculated (per node)

Fresh insights are computed from current response sample + topics + metrics, conditioned on prior_insights text — not checkpoint blobs.

NodeInputsOutputs / side effects
ingestDB responses (stratified sample); bootstrap flagresponses, new_response_ids, prior_snapshots
metricsAll loaded responsesNPS/CSAT/CES, compute_response_trend_analysis() → anomaly flag candidate
absa → cluster → topicsnew_response_ids (capped at 50 for incremental)Clusters, topic_signals, DB topic upserts
narrateMetrics, clusters, prior_insights from anchorL1–L3 insight dicts; or cache hit → reuse active insights
report_agenttopic_signals, prior_insights, _compute_delta_signals()report.* narrative insights (tiered)
verify / evaluateInsight set + citationsTrust scores, drop redundant insights
publishFinal insight listSupersede + INSERT insights; snapshot; checkpoint; Redis L3 facts

_compute_delta_signals(state) — report delta (not checkpoint delta)

Deterministic pre-LLM signals in crystalos/agents/tiered_report.py:

  • new_findings — topics where ≥15% of response IDs overlap new_response_ids
  • divergences — keyword match prior_insights headlines vs current topic_signals sentiment
  • trends — compare current metrics vs prior_snapshots

Uses state.prior_insights (from anchor run), not survey_insight_checkpoints.

Cache hit path (automated only)

if not force_regenerate
   and len(new_response_ids) == 0
   and current_insight_rows exist:
     return cached insights (insights_from_cache = true)
# verify/evaluate skip on cache hit
# publish STILL runs → still writes snapshot + checkpoint

To wire checkpoints into future insight runs (not implemented)

  1. In node_publish: load prior checkpoint row + read_checkpoint_blob(report_url); call compute_delta(); store in delta_from_prior and blob.
  2. In node_ingest or node_narrate: optionally load last checkpoint blob for structured prior metrics (today: snapshots suffice).
  3. Gate checkpoint INSERT on meaningful change or CHECKPOINT_FULL_RESPONSE_THRESHOLD.
  4. Fix trigger CHECK to allow schedule.

20. Run sequences (stream vs manual)

Automated stream run (+10 new responses)

response_stream._trigger_insights
  INSERT agent_runs (trigger_type set later, intent='insight:stream')
  POST /insights/generate { trigger: "stream" }

node_ingest     → load sample, new_response_ids = unenriched IDs (may be >0)
node_narrate    → anchor = last stream/schedule run OR last run with new_response_count>0
                → prior_insights from anchor.run_id → LLM context
                → may cache-hit if new_response_ids empty
node_report     → skip if <25 new since last report.*
node_publish    → supersede insights, INSERT new rows
                → INSERT survey_metric_snapshots (anomaly_flag from trend analysis)
                → write_checkpoint_blob + INSERT survey_insight_checkpoints
                → UPDATE agent_runs (new_response_count, prior_context_run_id)

Manual run (Generate button)

POST /api/insights/:surveyId/trigger
  INSERT agent_runs (intent='manual_refresh')
  POST /insights/generate { trigger: "manual" }

node_ingest     → force_regenerate=true → bootstrap sample size
node_narrate    → never cache-hit; full re-narration
                → prior_insights still from SAME anchor unless new_response_count>0 at publish
node_report     → always runs (manual bypasses 25-response gate)
node_publish    → same checkpoint path as stream
                → if new_response_ids was empty: anchor unchanged for NEXT run
Summary for engineers: Checkpoints are a publish-side artifact synchronized with insights/snapshots. Insight evolution is driven by anchor-run insights rows + survey_metric_snapshots + new_response_ids — not by reading prior checkpoints back into the graph.
